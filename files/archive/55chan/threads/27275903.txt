 >>27276137 \r\n >Modelagem no caso eu quero dizer se eu criar uma tabela irá depender de outra, chave estrangeira e chave primária \r\n Isso é puramente lógica, você só precisa entender como funciona um relacionamento e então praticar. \r\n >chave primária é uma coluna, como todas as outras, mas ela serve para identificação \r\n >ex: O CPF pode ser chave primária de uma tabela de 'Clientes', pois o CPF é único \r\n >a chave estrangeira é somente a chave primária de outra tabela \r\n >a chave estrangeira serve para 'ligar' uma tabela com a outra \r\n >ex: Tabela 'Clientes' e 'Enderecos', para saber qual endereço do cliente, você precisa 'liga-las' de alguma forma e essa forma é por chave estrangeira, ou seja você cria uma coluna para referenciar outra tabela \r\n >mas em qual tabela vai a chave estrangeira? \r\n >na tabela que tem possibilidade de dados múltiplos (no caso de 1:N), na de maior importância (no caso de 1:1) ou criar uma tabela nova para colocar ambas (no caso de N:N) \r\n >ex. (1:N): Um cliente possui dados pessoais únicos (tabela 'Clientes'), mas possuí um endereço do trabalho e outro de sua casa (tabela 'Endereco'), então a chave deve ficar no 'Endereco'. \r\n >ex. 2 (1:1): Seguindo o mesmo exemplo anterior, caso seja permitido somente um endereço por cliente, a chave estrangeira pode ficar na 'Clientes' (caso ocorra na prática não recomendo que faça isso, é somente para te explicar) \r\n >ex. 3 (N:N): Tabela 'Produtos' e 'Vendas', um produto pode estar em várias vendas, assim como uma venda pode possuir vários produtos, então para resolver esse problema você cria uma tabela intermediaria para 'liga-los', no nosso caso 'Carrinhos', onde você coloca a chave primária de ambos \r\n \r\n >1:1: Um para um \r\n >1:N: Um para vários \r\n >N:N: Vários para vários \r\n \r\n Em todo caso, um vídeo de prático que você pode acompanhar.
